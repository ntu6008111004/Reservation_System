## ภาคสรุปวิเคราะห์ตั้งแต่เริ่มออกแบบระบบ

### เป้าหมายหลักของระบบ

ระบบนี้ถูกออกแบบให้เป็นระบบจองห้องประชุมภายในองค์กรที่ใช้งานง่ายมากที่สุด โดยไม่บังคับให้ผู้ใช้ทั่วไปต้อง login หรือจำรหัสผ่าน เพราะกลุ่มผู้ใช้มีทั้งคนที่ถนัดเทคโนโลยีและผู้ใช้วัย 50-60 ที่ต้องการหน้าจอชัด กดง่าย และทำงานสำเร็จเร็ว

แนวคิดสำคัญ:
- ผู้ใช้ทั่วไปต้องจองได้เร็ว ไม่ต้องเรียนรู้ระบบเยอะ
- Admin ต้องควบคุมระบบได้หลัง login เท่านั้น
- ระบบต้องใช้ Google Apps Script + Google Sheet ให้คุ้ม แต่ต้องกันข้อจำกัดของ Sheet/Calendar/Apps Script ไว้ล่วงหน้า
- ถ้า feature ใดอาจรบกวนผู้ใช้ เช่น อีเมลแจ้งเตือนเต็มรูปแบบ ให้เตรียมโครงไว้ก่อน แต่ยังไม่เปิดใช้งานจริง

### เหตุผลที่เลือก Google Apps Script และ Google Sheet

Google Apps Script เหมาะกับงานนี้เพราะ:
- ใช้งานร่วมกับบัญชี Google เดิมขององค์กรได้ง่าย
- ไม่ต้องตั้ง server แยก
- deploy เป็น Web App ได้เร็ว
- เชื่อม Google Calendar และ Google Meet ได้โดยตรง
- Google Sheet เหมาะกับทีมที่ต้องการเปิดดูข้อมูลจริงแบบตาราง

ข้อจำกัดที่วิเคราะห์ไว้:
- Google Sheet จะเริ่มช้าถ้าอ่านข้อมูลทั้ง sheet จำนวนมาก
- Apps Script มี quota และ timeout
- Calendar API อาจสร้าง Meet สำเร็จแต่ขั้นตอนอื่นพลาดได้
- ถ้า row ใน Sheet เต็ม ระบบ append ข้อมูลจะมีปัญหา

สิ่งที่ทำเพื่อรองรับข้อจำกัด:
- เพิ่ม `booking_index` เพื่อแยกข้อมูลที่ใช้ตรวจเวลาซ้อน
- เพิ่ม auto-expand row เมื่อเหลือ row น้อยกว่าหรือเท่ากับ 100
- เพิ่ม Diagnostics ให้ตรวจ schema, Calendar, Meet, sheet capacity และระบบอื่น
- เพิ่ม QA script สำหรับตรวจสิ่งที่เคยเป็นปัญหา

จากการวิเคราะห์ผู้ใช้ภายในองค์กรและกลุ่มผู้ใช้วัย 50-60 ควรยังไม่เปิดอีเมลแจ้งเตือนเต็มรูปแบบทันที เพราะอาจทำให้ผู้ใช้รู้สึกว่าระบบซับซ้อนหรือมีข้อความรบกวน

สิ่งที่ตัดสินใจ:
- เตรียม `EmailNotificationService` ไว้
- ค่าเริ่มต้น `email_notifications_enabled=false`
- ถ้าผู้จองไม่กรอกอีเมล ระบบข้ามให้อัตโนมัติ
- ถ้าอนาคตเปิดส่งอีเมลแล้วเกิด error จะไม่ทำให้ booking/cancel ล้ม

เหตุผล:
- รองรับอนาคตโดยไม่รบกวนการใช้งานช่วงแรก
- ให้ผู้ใช้คุ้นกับการจองก่อน
- ค่อยเปิดอีเมลเมื่อ flow หลักนิ่งและคนใช้ชินมือแล้ว

### แนวคิดเรื่องความเร็วของระบบ

ระบบปัจจุบันทำงานได้ แต่ถ้าข้อมูลโตมากขึ้น จุดที่ต้องระวังคือการอ่านข้อมูลจาก Google Sheet ทั้ง sheet

จุดที่อาจช้าขึ้นเมื่อข้อมูลเยอะ:
- `DatabaseService.listObjects`
- `BookingIndexService.hasConflict`
- Dashboard ที่อ่าน bookings ทั้งหมด
- Audit Log ที่อ่านทั้งหมดแล้ว sort

สิ่งที่ทำแล้ว:
- เพิ่ม `booking_index` แยกข้อมูลตรวจ conflict
- จำกัด list booking แสดง 200 รายการ
- เพิ่ม auto-expand row
- เพิ่ม QA check และ Diagnostics

สิ่งที่ควรทำต่อ:
- อ่านข้อมูลเฉพาะช่วงวันที่ เช่น 30/60/90 วัน
- Audit Log ควรอ่านจากท้าย sheet แทนอ่านทั้งหมด
- Dashboard ควรมี summary/cache
- ใช้ CacheService กับ rooms/settings/dashboard
- เพิ่ม API สำหรับ availability check ที่คืนเฉพาะห้องว่างตามช่วงเวลา

### แนวคิดเรื่องความปลอดภัยและความง่าย

ระบบเลือกแนวทาง “ผู้ใช้ทั่วไปไม่ต้อง login” เพราะเป็นระบบภายในและต้องการลด friction ให้มากที่สุด แต่ Admin ต้อง login ก่อนเห็นข้อมูลจัดการ

สิ่งที่ทำแล้ว:
- Admin panel ถูกซ่อนไว้ก่อน login
- Admin session มีอายุ 8 ชั่วโมง
- Audit Log บันทึก action สำคัญ
- API admin ต้องใช้ session token

สิ่งที่ควรทำต่อ:
- server-side logout
- ป้องกัน brute force login
- จำกัด action อันตรายด้วย confirm ที่ชัดขึ้น
- เพิ่ม role ในอนาคต เช่น SUPER_ADMIN, STAFF_ADMIN

### แนวคิดเรื่อง UX สำหรับผู้ใช้วัย 50-60

สิ่งที่ควรให้ความสำคัญ:
- ปุ่มใหญ่พอกดง่าย
- ตัวหนังสือไม่เล็กและไม่หนาจนอ่านยาก
- คำอธิบายสั้น ชัด ไม่ใช้ศัพท์เทคนิคมาก
- error message ต้องบอกว่าต้องแก้อะไร
- ไม่ให้ผู้ใช้ต้องจำขั้นตอนเยอะ
- ลด field ที่ไม่จำเป็น
- หลีกเลี่ยง AM/PM
- ใช้ modal สรุปหรือแจ้งผลแทนข้อความเล็ก ๆ ที่มุมจออย่างเดียว

สิ่งที่ทำแล้ว:
- เปลี่ยนเป็นฟอนต์ Kanit
- ใช้เวลา 24 ชั่วโมง
- วันที่แสดงแบบ `dd/mm/yyyy`
- ปรับ responsive design
- เพิ่ม modal/loading/result
- อีเมลไม่บังคับ

สิ่งที่ควรทำต่อ:
- เพิ่ม modal สรุปก่อนยืนยันจอง
- เพิ่มข้อความช่วยเหลือข้างช่องวันที่/เวลาแบบภาษาคนทั่วไป
- เพิ่มปุ่ม “จองอีกครั้ง” หลังจองสำเร็จ
- เพิ่มปุ่ม copy Google Meet/GPS
- เพิ่ม empty state ที่บอกว่าควรทำอะไรต่อ

### สรุปสิ่งที่ควรทำต่อเป็นลำดับ

1. เพิ่มเช็คห้องว่างทันทีหลังเลือกวันเวลา
2. เพิ่ม modal สรุปก่อนยืนยันจอง
3. เพิ่ม rollback กรณี Calendar สร้างสำเร็จแต่ Sheet บันทึกพลาด
4. เพิ่ม booking policy เช่น ห้ามย้อนหลัง, เวลาทำการ, ระยะเวลาขั้นต่ำ
5. เพิ่ม admin edit/reschedule booking
6. เพิ่ม performance optimization สำหรับข้อมูลจำนวนมาก
7. เพิ่ม server-side logout และ login rate limit
8. เพิ่ม Audit Log filter
9. เปิด email notification เฉพาะเมื่อผู้ใช้ชินกับระบบแล้ว

### บทสรุปการออกแบบ

ระบบนี้ควรเดินแนวทาง “ง่ายก่อน ฉลาดขึ้นทีหลัง” คือให้ผู้ใช้จองได้สำเร็จโดยไม่ต้องคิดเยอะ แล้วค่อยเพิ่ม automation และ notification เมื่อ flow หลักนิ่งแล้ว การเลือกไม่บังคับ login และไม่บังคับอีเมลเป็นแนวทางที่เหมาะกับระบบภายในองค์กร ส่วนฝั่ง Admin ต้องค่อย ๆ เพิ่มเครื่องมือควบคุมเพื่อให้ดูแลระบบได้โดยไม่ต้องแก้ Google Sheet เอง
