# คู่มือใช้งานและคู่มือทดสอบระบบจองห้องประชุม

เอกสารนี้ใช้สำหรับน้อง IT ทดสอบระบบ สอนผู้ใช้งาน และใช้เป็น checklist ก่อนส่งใช้งานจริง

## 1. ภาพรวมระบบ

ระบบนี้เป็น Web App บน Google Apps Script สำหรับจองห้องประชุมภายในองค์กร โดยใช้ Google Sheet เป็นฐานข้อมูล และเชื่อม Google Calendar / Google Meet สำหรับการประชุมออนไลน์

ผู้ใช้ทั่วไป:
- ไม่ต้อง login
- เลือกรูปแบบประชุม
- เลือกห้องหรือทรัพยากร
- กรอกชื่อผู้จอง
- เลือกวันที่และเวลา
- กดจอง

ผู้ดูแลระบบ:
- ต้อง login
- ดู Dashboard
- ดูและยกเลิกรายการจอง
- เปิด/ปิดทรัพยากร
- ดู Audit Log
- ตรวจ Diagnostics
- ตั้งค่าฟีเจอร์ที่เตรียมไว้ในอนาคต
- ล้างข้อมูลจองหลังทดสอบ

## 3. โครงสร้างหน้าจอหลัก

### 3.1 จองห้อง

ใช้สำหรับสร้างรายการจองใหม่

ข้อมูลที่ผู้ใช้ต้องกรอก:
- รูปแบบประชุม
- ห้องประชุมหรือทรัพยากร
- หัวข้อประชุม
- ชื่อผู้จอง
- วันที่เริ่ม
- เวลาเริ่ม
- วันที่สิ้นสุด
- เวลาสิ้นสุด

ข้อมูลที่ไม่บังคับ:
- อีเมลผู้จอง
- หมายเหตุ
- GPS / Google Maps สำหรับประชุมนอกสถานที่

### 3.2 รายการจอง

ใช้ดูรายการจองแบบ Card และ Table

ตัวกรองที่มี:
- ห้องประชุม
- รูปแบบประชุม
- ตั้งแต่วันที่
- ถึงวันที่
- ค้นหาจากหัวข้อ ผู้จอง อีเมล หรือชื่อห้อง

### 3.3 ปฏิทิน

ใช้ดูรายการจองแบบแยกตามวัน

เหมาะสำหรับ:
- ดูภาพรวมว่าวันนี้มีจองอะไร
- ดูช่วงเวลาของแต่ละรายการ
- ดูประเภทประชุม

### 3.4 ผู้ดูแลระบบ

ต้อง login ก่อนจึงเห็นข้อมูลจัดการ

ส่วนที่มี:
- Dashboard
- จัดการรายการจอง
- จัดการทรัพยากร
- ตั้งค่าฟีเจอร์
- Audit Log
- Diagnostics
- Maintenance

## 5. รูปแบบประชุม

### 5.1 ประชุมในห้อง

ใช้สำหรับห้องประชุมจริง เช่น Meeting Room 101

สิ่งที่ระบบทำ:
- แสดงเฉพาะห้องจริง
- ตรวจเวลาซ้อน
- บันทึกลง Google Sheet

### 5.2 ออนไลน์

ใช้สำหรับประชุมผ่าน Google Meet

สิ่งที่ระบบทำ:
- แสดง resource สำหรับ Online Meeting
- สร้าง Google Calendar event
- สร้าง Google Meet URL
- เจ้าของระบบเป็นผู้สร้าง Meet หลัก

หมายเหตุ:
- ตอนนี้ฟีเจอร์เชิญอีเมลผู้จองเข้า Calendar/Meet ถูกปิดไว้ก่อน
- Admin สามารถเปิดได้ในอนาคตจากหน้า `ตั้งค่าฟีเจอร์`

### 5.3 นอกสถานที่

ใช้สำหรับบันทึกการประชุมนอกสำนักงาน

สิ่งที่ระบบทำ:
- แสดง resource นอกสถานที่
- เปิดช่อง GPS / Google Maps
- ไม่สร้าง Google Meet

## 6. การใช้งานหน้า Admin

### 6.1 Login

1. ไปที่แท็บ `ผู้ดูแลระบบ`
2. กรอกอีเมลผู้ดูแล
3. กรอกรหัสผ่าน
4. กด `เข้าสู่ระบบ`

หลัง login สำเร็จ ระบบจะแสดง:
- Dashboard
- รายการจอง
- ทรัพยากร
- ตั้งค่าฟีเจอร์
- Audit Log
- Diagnostics
- Maintenance

### 6.2 Dashboard

ใช้ดูภาพรวมระบบ:
- จำนวนจองทั้งหมดที่ยังใช้งาน
- Online
- Onsite
- นอกสถานที่
- รายการที่ยกเลิกแล้ว
- จำนวนทรัพยากรที่ active
- รายการจองล่าสุด

### 6.3 จัดการรายการจอง

ใช้ดูรายการจองทั้งหมด รวมถึงรายการที่ถูกยกเลิก

การยกเลิก:
1. ไปที่ `จัดการรายการจอง`
2. กด `โหลดรายการจอง`
3. เลือกรายการที่ต้องการ
4. กด `ยกเลิก`
5. ยืนยัน

ผลลัพธ์:
- booking เปลี่ยนสถานะเป็น `CANCELLED`
- ช่วงเวลานั้นถูกปล่อยให้จองใหม่ได้
- ถ้ามี Calendar event ระบบพยายามลบ event นั้น
- Audit Log บันทึก action

### 6.4 จัดการทรัพยากร

ใช้เปิด/ปิดห้องหรือ resource

ตัวอย่าง:
- ปิดห้องที่ซ่อมอยู่
- เปิด resource กลับมาเมื่อใช้งานได้

ผลลัพธ์:
- ถ้าปิด resource ผู้ใช้ทั่วไปจะไม่เห็น resource นั้นตอนจอง
- Audit Log บันทึก action

### 6.5 ตั้งค่าฟีเจอร์

ใช้เปิด/ปิดฟีเจอร์ที่เตรียมไว้ในอนาคต

ฟีเจอร์ที่มีตอนนี้:

#### ส่งอีเมลแจ้งเตือนการจอง/ยกเลิก

ค่าเริ่มต้น:

```text
ปิด
```

เหตุผล:
- ต้องการให้ผู้ใช้ภายใน โดยเฉพาะผู้ใช้วัย 50-60 ใช้ระบบให้ชินก่อน
- ลดความสับสนจากอีเมลแจ้งเตือน

ถ้าเปิดในอนาคต:
- ระบบจะส่งอีเมลเฉพาะคนที่กรอกอีเมล
- ถ้าไม่กรอกอีเมล ระบบจะข้าม

#### เชิญอีเมลผู้จองเข้า Google Meet

ค่าเริ่มต้น:

```text
ปิด
```

พฤติกรรมเมื่อปิด:
- เจ้าของระบบยังเป็นผู้สร้าง Calendar/Meet หลัก
- ถ้าผู้ใช้กรอกอีเมล ระบบยังไม่เชิญอีเมลนั้นเข้า event

พฤติกรรมเมื่อเปิด:
- เจ้าของระบบยังเป็นผู้สร้าง Calendar/Meet หลัก
- ถ้าผู้ใช้กรอกอีเมล ระบบจะเชิญอีเมลนั้นเป็นผู้ร่วมประชุม

วิธีเปิด/ปิด:
1. Login Admin
2. ไปที่ `ตั้งค่าฟีเจอร์`
3. ติ๊กหรือเอาติ๊กออก
4. กด `บันทึกการตั้งค่า`
5. ตรวจ Audit Log ว่ามี action `FEATURE_SETTINGS_UPDATED`

### 6.6 Audit Log

ใช้ดูประวัติ action สำคัญ เช่น:
- Admin login
- สร้าง booking
- ยกเลิก booking
- เปิด/ปิด resource
- อัปเดต feature settings
- ล้างข้อมูลจอง
- API error

### 6.7 Diagnostics

ใช้ตรวจระบบหลัง deploy หรือเมื่อระบบมีปัญหา

สิ่งที่ตรวจ:
- Script Properties
- Spreadsheet ID
- Database schema
- Settings
- Email notification status
- Sheet capacity
- Calendar access
- Google Meet creation
- Cache service
- Lock service
- Admin exists
- Web App config

### 6.8 Maintenance

ใช้ล้างข้อมูลจองหลังทดสอบ

สิ่งที่ลบ:
- bookings
- booking_index

สิ่งที่ไม่ลบ:
- admin
- rooms
- settings
- schema
- audit_logs

วิธีใช้:
1. Login Admin
2. ไปที่ `Maintenance`
3. กด `ล้างข้อมูลจอง`
4. อ่านคำเตือน
5. กดยืนยัน

## 10. ปัญหาที่อาจเจอและวิธีแก้

### 10.1 จองออนไลน์แล้วไม่มี Meet URL

ให้ตรวจ:
- Calendar API เปิดแล้วหรือยัง
- Apps Script มี scope Calendar หรือไม่
- Diagnostics `calendarMeetCreation` ผ่านหรือไม่
- เจ้าของระบบมีสิทธิ์สร้าง event ใน Calendar หรือไม่

### 10.2 ผู้ใช้บอกเลือกวันที่ไม่ได้

ให้ตรวจ:
- ใช้ date picker custom ของระบบ ไม่ใช่ native browser
- Console ไม่มี error เกี่ยวกับ `showPicker`
- QA script ผ่านหรือไม่

### 10.3 รายการจองไม่แสดง

ให้ตรวจ:
- รายการนั้น status เป็น `CONFIRMED` หรือไม่
- ถ้าถูกยกเลิก จะไม่แสดงในหน้าผู้ใช้
- Admin ยังดูได้ใน Booking Management

### 10.4 Sheet row ใกล้เต็ม

ระบบจะเพิ่ม row ให้อัตโนมัติเมื่อเหลือน้อยกว่าหรือเท่ากับ 100 rows

ให้ตรวจ:
- Admin > Diagnostics > `sheetCapacity`

### 10.5 ระบบช้า

อาจเกิดเมื่อข้อมูลใน Google Sheet เยอะมาก

แนวทางปรับปรุงในอนาคต:
- จำกัดช่วงวันที่ในการดึง booking
- cache rooms/settings/dashboard
- อ่าน audit log จากท้าย sheet
- ทำ summary sheet สำหรับ dashboard

## 12. ฟีเจอร์ที่เตรียมไว้แต่ยังไม่ควรเปิดทันที

### 12.1 ส่งอีเมลแจ้งเตือน

สถานะปัจจุบัน:

```text
ปิด
```

เหตุผล:
- รอให้ผู้ใช้คุ้นกับระบบก่อน
- ลดความสับสน
- ลดอีเมลรบกวน

### 12.2 เชิญอีเมลผู้จองเข้า Google Meet

สถานะปัจจุบัน:

```text
ปิด
```

เหตุผล:
- เจ้าของระบบยังเป็นผู้สร้าง Meeting หลัก
- เตรียมไว้สำหรับอนาคต
- เปิดได้เมื่อองค์กรต้องการให้ผู้จองได้รับ invite โดยตรง

## 13. สิ่งที่ควรพัฒนาต่อ

ลำดับแนะนำ:
1. เช็คห้องว่างทันทีหลังเลือกวันเวลา
2. เพิ่ม modal สรุปก่อนยืนยันจอง
3. เพิ่ม rollback ถ้า Calendar สร้างสำเร็จแต่ Sheet บันทึกพลาด
4. เพิ่ม admin edit/reschedule booking
5. เพิ่ม filter ใน Audit Log
6. เพิ่ม server-side logout
7. เพิ่ม login rate limit
8. เพิ่ม cache/performance optimization เมื่อข้อมูลเยอะ

## 14. สรุปสำหรับ IT

ระบบนี้พร้อมสำหรับทดสอบใช้งานภายในระดับแรกแล้ว โดยเน้นความง่ายสำหรับผู้ใช้ทั่วไปและมีเครื่องมือให้ Admin ดูแลระบบได้จากหน้าเว็บ

สิ่งสำคัญก่อนปล่อยจริง:
- ทดสอบจองทั้ง 3 รูปแบบ
- ทดสอบ Google Meet
- ทดสอบ Admin cancel
- ทดสอบเปิด/ปิด resource
- ทดสอบ Diagnostics
- ทดสอบมือถือ
- ล้างข้อมูลจองทดสอบก่อนส่งใช้งานจริง หากต้องการเริ่มข้อมูลใหม่
